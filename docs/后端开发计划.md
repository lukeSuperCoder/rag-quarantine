# 智能问答助手后端开发计划

## 1. 开发目标

基于现有脚本能力，开发 Python + Flask 后端服务，为前端提供文档上传、知识库管理、RAG 问答、流式输出、多轮会话和引用溯源接口。

现有脚本已经覆盖文档转换、分块、Milvus collection 创建、embedding 写入、检索、rerank 和 RAG 对话。后端开发的重点不是重写算法，而是把脚本能力服务化、模块化和持久化。

## 2. 技术选型

| 模块 | 选型 | 说明 |
|------|------|------|
| Web 框架 | Flask | 提供 REST API 和 SSE 流式接口 |
| 跨域 | flask-cors | 支持前端本地开发联调 |
| 本地状态库 | SQLite | 保存文档、chunk、会话、消息、任务状态 |
| 向量库 | Milvus Lite | 复用当前 `milvus_quarantine.db` |
| Embedding | 智谱 `embedding-3` | 复用当前配置 |
| Chat Model | 智谱 GLM | 复用当前 `06_chat.py` 模型调用 |
| Rerank | 智谱 rerank API | 保留分数退化兜底 |

## 3. 后端目录规划

建议新增 `backend/` 目录，保留 `scripts/` 作为可独立运行的 CLI 工具。

```text
backend/
├── app.py                    # Flask 入口
├── api/
│   ├── chat.py               # 会话和流式问答接口
│   ├── documents.py          # 文档上传和管理接口
│   ├── knowledge.py          # 知识库查询和片段管理接口
│   └── health.py             # 健康检查
├── services/
│   ├── document_service.py   # 文档保存、转换、状态流转
│   ├── chunk_service.py      # 文本分块
│   ├── vector_service.py     # embedding、Milvus 写入、检索、删除
│   ├── rerank_service.py     # 重排序与兜底
│   ├── rag_service.py        # RAG 问答编排
│   └── storage_service.py    # SQLite 读写
├── models/
│   └── schemas.py            # API 请求/响应结构定义
└── db/
    ├── init_db.py            # 初始化 SQLite 表
    └── app.sqlite3           # 本地持久化数据库
```

## 4. 数据库设计

### 4.1 documents 表

保存上传文档和内置文档的处理状态。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT | 文档 ID，UUID |
| filename | TEXT | 原始文件名 |
| doc_name | TEXT | 文档名称 |
| doc_type | TEXT | 文档类型 |
| country | TEXT | 国家或地区 |
| source_path | TEXT | 原始文件路径 |
| txt_path | TEXT | 转换后 txt 路径 |
| status | TEXT | 上传和处理状态 |
| chunk_count | INTEGER | 分块数量 |
| error_message | TEXT | 失败原因 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |

状态流转：

```text
uploaded -> converting -> chunking -> embedding -> ready
                                      -> failed
```

### 4.2 chunks 表

保存 chunk 元数据和正文，用于关键词查询、引用展示和 Milvus 删除映射。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT | chunk ID，UUID |
| document_id | TEXT | 所属文档 ID |
| milvus_id | INTEGER | Milvus 主键，如果可获取 |
| text | TEXT | chunk 正文 |
| doc_name | TEXT | 文档名称 |
| doc_type | TEXT | 文档类型 |
| country | TEXT | 国家或地区 |
| chunk_index | INTEGER | 文档内分块序号 |
| section_title | TEXT | 章节标题 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |

建议在 Milvus schema 中新增 `document_id` 和 `chunk_id` 字段，便于按文档删除和引用定位。新增字段后需要重建 collection 并重新写入现有知识库。

### 4.3 chat_sessions 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT | 会话 ID |
| title | TEXT | 会话标题 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |

### 4.4 chat_messages 表

| 字段 | 类型 | 说明 |

## 5. 服务模块设计

### 5.1 document_service

职责：

- 接收上传文件并保存到本地上传目录。
- 判断文件类型，只允许 `.doc`、`.docx`、`.xls`。
- 调用现有 `01_convert_docs.py` 中的转换逻辑。
- 记录每个处理阶段的状态和错误信息。

关键规则：

- 原始 `政策法规文档/` 仍作为初始数据输入，不在 API 上传时覆盖。
- 用户上传文件存放到 `data/uploads/`。
- 转换后的 txt 存放到 `data/txt/` 或 `data/upload_txt/`。

### 5.2 chunk_service

职责：

- 复用 `02_chunk_documents.py` 的文档类型判断和分块策略。
- 对上传文档支持用户传入 `doc_type` 和 `country`。
- 输出统一 chunk 字段：`text`、`doc_name`、`doc_type`、`country`、`chunk_index`、`section_title`。

关键规则：

- 不破坏现有分块元数据契约。
- 对未知文档类型使用通用段落分块。

### 5.3 vector_service

职责：

- 调用智谱 embedding API 批量生成向量。
- 将 chunk 写入 Milvus。
- 执行向量检索和元数据过滤。
- 删除指定文档或指定 chunk 的向量。
- 处理 Milvus 连接和 collection 初始化。

关键规则：

- 使用 `VECTOR_DIM = 2048`，保持与 `embedding-3` 一致。
- 写入 Milvus 时同时保存 `document_id`、`chunk_id`、`doc_name`、`doc_type`、`country`、`chunk_index`、`section_title`。
- 对 embedding 失败的文本不写入零向量，应标记任务失败并记录原因，避免污染知识库。

### 5.4 rerank_service

职责：

- 对向量检索候选结果调用 rerank。
- 为 rerank 输入补充文档名、类型、国家、章节标题。
- 当 rerank 返回异常、缺少分数或分数区分度过低时，回退原始 Milvus 排序。

该策略沿用当前 `scripts/06_chat.py` 中已经验证过的兜底逻辑。

### 5.5 rag_service

职责：

- 根据用户问题执行 embedding、Milvus 检索、可选 rerank。
- 拼接带引用的上下文。
- 读取当前 session 最近 N 轮历史消息。
- 调用模型生成回答。
- 支持 SSE 流式输出。
- 保存用户消息、助手消息和引用来源。

Prompt 要求：

- 必须基于参考资料回答。
- 参考资料不足时明确说明。
- 涉及知识库事实时必须引用政策文件名称和章节。
- 回答简洁准确，优先给出结论。

## 6. API 开发阶段

### 第一阶段：后端基础骨架

- 新增 Flask app。
- 增加健康检查接口。
- 增加 SQLite 初始化脚本。
- 增加统一错误响应格式。
- 配置 CORS，支持前端本地开发。

### 第二阶段：RAG 服务 API

- 实现会话创建、会话列表、消息列表。
- 实现非流式 RAG 问答用于调试。
- 实现 SSE 流式问答。
- 验证多轮历史和引用来源保存。

### 第三阶段：文档处理 API

- 实现文档上传。
- 实现转换、分块、embedding、入库流程。
- 实现文档列表和处理状态查询。
- 实现失败状态和错误信息记录。

### 第四阶段：知识库管理 API

- 实现关键词查询。
- 实现文档删除，并同步删除 Milvus 数据。
- 实现 chunk 查看。
- 可选实现 chunk 编辑和重新 embedding。

### 第五阶段：稳定性与验证

- 补充参数校验。
- 补充关键服务的单元级检查脚本。
- 使用真实问题验证 RAG、rerank 和引用输出。
- 与前端完成端到端联调。

## 7. 验收标准

- Flask 服务可以启动并通过健康检查。
- 可以创建会话、发送问题、接收流式回答。
- 同一会话中追问可以携带历史上下文。
- 回答结果包含可结构化展示的引用来源。
- 可以上传文档并完成转换、分块、embedding、入库。
- 可以查询文档列表和知识库片段。
- 删除文档后，Milvus 检索不再返回该文档对应内容。
- rerank 异常或分数退化时不会降低回答质量，会回退原始向量排序。
